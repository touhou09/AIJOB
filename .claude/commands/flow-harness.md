# Flow Harness 실행

사용자 요청을 AIJOB 업무 Flow Harness로 정규화한다.

## 사용 시점

- 새 M+ 업무를 시작할 때
- 요구사항/범위/검증 기준이 대화로만 흩어져 있을 때
- 외부 팀/운영/리뷰어에게 공유해야 하는 업무일 때
- 구현 이후 테스트 기록/결과 보고/티켓 trace가 필요한 업무일 때

## 절차

### 1. 요구사항 카드 작성

대화, STATE.md, TODO.md, 관련 work/ 파일, 관련 티켓/문서에서 아래를 채운다.

- 제목
- 배경
- 원하는 최종 상태
- 포함/제외 범위
- 제약
- 완료 조건
- 검증 증거
- flow 단계
- tool/채널
- 공유 대상

### 2. 업무 정의서 draft 작성

`.claude/runbooks/flow-harness-templates.md`의 업무 정의서 템플릿을 사용한다.

정의서에는 반드시 다음이 들어간다.

- 한 줄 정의
- 배경/목표/범위
- 업무 Flow
- Tool/채널 선정 이유
- 역할/리뷰
- 완료 조건
- 검증 계획
- 공유/보고 계획
- 리스크/롤백

### 3. 실행 전 공유

외부 영향이 있으면 Confluence 또는 티켓 코멘트에 정의서 링크를 공유한다.
작업 티켓이 있으면 착수 코멘트를 남긴다.

### 4. 구현/운영 조치 수행

실행 중 방향성 변경, 피벗, 개선사항 발견은 티켓 코멘트에 남긴다.

### 5. 구현 테스트 기록 작성

아래를 기록한다.

- 테스트 전략
- 테스트 환경
- 성공/실패 기준
- 실제 실행 기록
- 증거
- 테스트 피드백
- Not-tested
- 재테스트 필요 항목
- 최종 판정

### 6. 결과 보고서 작성

최종 결론은 다음 중 하나로 분류한다.

- 완료
- 부분 완료
- 보류
- 실패
- 피벗 필요

결과 보고서에는 요구사항 대비 결과와 검증 증거를 연결한다.

### 7. 티켓/문서 이관

- 결과 보고서 링크를 티켓 코멘트에 남긴다.
- 리뷰어/요청자를 멘션한다.
- 필요한 경우 상태를 Review로 전환하고 담당자를 요청자/리뷰어로 넘긴다.
- 후속 작업은 별도 티켓으로 분리한다.

## 출력

마지막 응답에는 다음만 간결히 보고한다.

- 정의서 위치
- 테스트 기록 위치
- 결과 보고서 위치
- 티켓 코멘트/상태 처리 결과
- 남은 리스크/후속
