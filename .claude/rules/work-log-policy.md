# 작업 기록 정책

## 문서화 대상
- M 이상 (2일 이상 소요) 작업, 아키텍처 변경, 외부 의존성 추가, 트레이드오프가 존재하는 결정
- 단순 버그 픽스, 스타일 변경, 라이브러리 버전 업데이트는 커밋 메시지로 충분 — 문서화 금지

## 템플릿

```markdown
## YYYY-MM-DD: 작업 제목 [done|in-progress|blocked]
- **What**: 한 줄 변경 요약
- **Why**: 결정 이유 (대안이 있었다면: X 대신 Y, 이유)
- **Impact**: 영향 범위 (누가, 어떤 시나리오)
- **Test**: 통과/실패 요약 + 커버 못하는 영역
- **Trap**: 시도→실패→해결 (30분 이상 삽질한 것만, 없으면 생략)
- **Next**: 다음 단계 또는 블로커 (done이면 생략 가능)
- **[DEBT]**: 수용한 기술 부채 (있을 때만, 정리 시 DECISIONS.md로 이동)
---
```

## 필드 규칙
- **What**: 코드에서 유추 불가능한 것만. "파일 X 수정" 같은 건 금지
- **Why**: 가장 중요한 필드. "왜 이 방식을 선택했는가"에 답해야 함
- **Impact**: 한 줄. "전체 유료 사용자 영향" 수준
- **Test**: "통과"보다 "커버 못하는 영역"이 더 중요
- **Trap**: 디버깅 30분 이상 소요한 것만. 증상→원인→해결 3단 구조
- **[DEBT]**: 세션 정리 시 DECISIONS.md로 이동 후 work에서 삭제

## 기록 금지 항목
- 변경 파일 경로 나열 (git log --stat으로 확인)
- 코드 스니펫 복사
- 의존성 버전 정보 (package.json이 source of truth)
- 소요 시간, AI 모델 버전
- 감정적 표현, 회의록 스타일 서술

## 크기 제한
- 항목당 6-9줄 (Trap, DEBT는 있을 때만)
- work/ 파일당 200줄 한도
- 150줄 초과 시 정리 트리거
