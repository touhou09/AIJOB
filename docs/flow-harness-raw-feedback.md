# AIJOB 업무 Flow Harness Raw Feedback

> 여기에 사용자가 말하는 피드백을 원문에 가깝게 누적하고, 아래 draft/spec로 정제한다.

## 2026-04-30

- 업무 flow 관련 개선사항이 필요하다는 피드백 수신.
- 사용자는 대화로 말하는 내용을 기반으로 AIJOB 기반 하네스를 만들고 싶어함.

### 피드백 #1 — 요구사항 기반 업무 flow 설계/공유

- 일이 들어오면 먼저 요구사항을 정의해야 한다.
- 정의한 요구사항에 맞춰 원하는 flow 또는 구조화된 업무 flow를 만든다.
- 그 과정에서 필요한 tool, 문서, 티켓, 관측 수단 등을 선정한다.
- 선정한 flow/tool/구조를 문서화하여 모두에게 공유해야 한다.
- 즉, 실행부터 들어가는 것이 아니라 “요구사항 → flow 설계 → tool 선정 → 공유 문서화”가 첫 번째 표준 업무 flow가 되어야 한다.

### 피드백 #2 — 업무 정의서 작성/공유

- 요구사항과 설계한 flow/tool 선정 내용을 바탕으로 업무 정의서를 작성해야 한다.
- 업무 정의서는 “어떤 업무를 진행할 것인지”를 모두가 이해할 수 있게 공유하는 문서여야 한다.
- 실행 전 팀 공유를 목적으로 하며, 작업 범위/완료 조건/검증 방식/역할/산출물을 명확히 해야 한다.

### 피드백 #3 — 구현 후 테스트/검증/피드백 기록

- 구현 작업 자체뿐 아니라, 구현 이후 이 구현을 어떻게 테스트할지 테스트 방식이 정의되어야 한다.
- 구현에 대한 검증 기록이 남아야 한다.
- 실제 테스트 실행 기록도 남아야 한다.
- 테스트 과정에서 나온 피드백 역시 기록으로 남아야 한다.
- 따라서 구현 완료 보고는 “코드가 됐다”가 아니라 “어떤 방식으로 검증했고, 어떤 결과/피드백이 남았는지”까지 포함해야 한다.

### 피드백 #4 — 작업 완료 결과 보고서/결론 공유

- 최종적으로 작업이 마무리되면 결과 보고서를 작성해야 한다.
- 결과 보고서는 모두와 공유되어야 한다.
- 현재 어떤 작업에 대해 어떤 결론이 나왔는지 명확히 공유되어야 한다.
- 결과 보고는 단순 완료 알림이 아니라 목표/수행/검증/피드백/결론/후속을 포함해야 한다.

### 피드백 #5 — 티켓 코멘트 기반 진행 흔적/방향 변경 기록

- 각 과정을 진행하는 동안 티켓 코멘트로 충분히 흔적을 남겨야 한다.
- 방향성 변경, 개선, 피벗, 범위 변경이 생기는 경우에도 코멘트로 남겨야 한다.
- 티켓 코멘트는 단순 완료 보고가 아니라 진행 과정의 trace 역할을 해야 한다.
- 누가 보더라도 티켓만 읽고 현재 어디까지 왔고 왜 방향이 바뀌었는지 알 수 있어야 한다.
