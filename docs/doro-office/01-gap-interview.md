# DOR-5 gap 재기획 인터뷰 시트

> 사용자가 "당초 생각한 구조와 다르다"고 한 지점을 체계적으로 끌어내기 위한 질문 리스트.
> 각 질문 아래에 사용자 답변을 받아서 채우고, 그 결과를 DOR-17 코멘트로 올리면 orchestrator가 분해 가능.

## A. 배포 토폴로지

1. 모니터링이 "스크립트"가 아니라 **상주 서버 프로세스**여야 한다고 생각했나? (예/아니오)
   - 예라면: 왜? (항상 응답 가능? WebSocket? 실시간 대시보드 지원?)
   - 아니오라면: 5분 간격 polling 주기가 적절한가, 더 짧아야 하나?

2. Mac Mini 한 대에서만 돌아도 충분한가, 아니면 **다중 호스트** (랩톱/리모트) 지원이 필요한가?

3. 모니터 자체가 죽으면 누가 알리나? (현재: 아무도. 사용자가 눈치채야 함)

## B. 사용자 인터페이스

4. 현재는 CLI (`/hermes-status`) + JSON 스냅샷만 있음. **웹 UI/대시보드**가 필요한가?
   - 필요하다면: 누가 보나? (본인만? 외부 공유? 모바일?)
   - 어떤 종류의 뷰? (실시간 스트림 vs 일일 요약 vs 히스토리 그래프)

5. Slack 알림 외에 **다른 채널**이 필요한가? (이메일/SMS/푸시/Telegram/전화)

6. Slack 채널 하나(`#sprint-planning`)로 충분한가, **심각도별/프로필별 분리**가 필요한가?

## C. 데이터와 히스토리

7. 현재는 최신 스냅샷 하나만 유지 (`latest.json` 덮어쓰기). **히스토리**가 필요한가?
   - 필요하다면: 며칠? 몇 개월?
   - 어디에? (JSON 파일 롤오버 / SQLite / 외부 DB / 시계열 TSDB)
   - 왜? (회고? 장기 트렌드? 성능 회귀 감지? 사후 분석?)

8. **어떤 메트릭을 시계열로 보고 싶나**?
   - 예: 에이전트별 일일 처리량, 평균 응답 지연, Slack 충돌 빈도, Paperclip 이슈 완료율
   - 또는 이슈별 처리 시간, 에이전트별 "말이 많은 정도" (코멘트 수)

## D. 감지와 알림

9. 현재 alert 조건 5종으로 충분한가? 추가로 감지해야 할 것이 있나?
   - 후보: 에이전트별 비용(budget) 초과, 이슈 쌓임(backlog depth), 특정 에이전트 연속 실패, LLM API 오류, Cloudflare tunnel 끊김

10. 알림 **threshold**가 지금은 하드코딩 (`30분`, `1500ms`). 사용자별/에이전트별 튜닝이 필요한가?

11. 같은 알림이 반복 전송되지 않도록 **억제/쿨다운** 로직이 필요한가? (현재 미구현)

12. 이상 탐지 시 **자동 복구**가 필요한가?
    - 예: gateway 죽으면 자동 재시작, heartbeat stale 에이전트 자동 wake, Slack 충돌 시 토큰 재할당
    - 아니면 단순 알림만?

## E. 에이전트와의 관계

13. 현재는 **모니터가 에이전트를 관찰**하는 방향. 반대로 **에이전트가 스스로 상태를 보고**하는 push 모델은 어떤가?
    - 장점: 실시간성, 에이전트가 내부 상태를 더 정확히 앎
    - 단점: 에이전트 SOUL.md에 보고 책임 추가, 중복 코드

14. 모니터 결과를 **에이전트가 스스로 사용**하는가? 예: orchestrator가 heartbeat stale 에이전트를 보고 재분배, CTO가 backlog 쌓이면 우선순위 조정

15. 모니터링 데이터가 `~/.hermes/memories/` 나 LLM Wiki에 **ingest**되어야 하나? (에이전트가 "지난주 내 처리량" 같은 걸 알아야 하나)

## F. 운영 측면

16. 현재 Slack webhook URL이 미설정 → 실제 알림 경로 검증이 안 됨. 이걸 먼저 챙길지?

17. launchd plist가 **예시 파일만** 존재하고 실제 등록 안 됨. 자동 부팅 시작을 어떻게 할지?

18. 모니터 설정을 바꿀 때 (임계값, 수집 주기 등) **어디를 건드리나**?
    - 현재: 코드 상수. 변경하려면 파일 편집 + 재시작
    - 후보: `~/.hermes/monitor.yaml` 전용 설정 파일, 환경변수, CLI flag 상시화

19. 모니터 자체의 **배포/업데이트**를 어떻게 할지? AIJOB 레포 pull → launchd 재시작?
